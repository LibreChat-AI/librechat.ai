LibreChat verfügt über eine robuste Streaming-Architektur, die sicherstellt, dass Sie niemals KI-generierte Inhalte verlieren. Egal, ob Ihre Verbindung unterbrochen wird, Sie den Tab wechseln oder auf einem anderen Gerät fortfahren – Ihre Antworten bleiben stets erhalten und synchronisiert.

## Warum das wichtig ist [#why-it-matters]

Herkömmliche Chat-Anwendungen verlieren alle Streaming-Inhalte, wenn die Verbindung abbricht. Mit fortsetzbaren Streams kann LibreChat:

- **Bewahrt jede Antwort** — Netzwerkprobleme, Browser-Aktualisierungen oder Server-Neustarts führen nicht zu Datenverlust
- **Hält mehrere Tabs synchron** — Öffnen Sie dieselbe Konversation in zwei Browser-Tabs und beobachten Sie, wie sie in Echtzeit gemeinsam aktualisiert werden
- **Ermöglicht nahtlosen Gerätewechsel** — Beginnen Sie eine Unterhaltung auf Ihrem Desktop und setzen Sie diese auf Ihrem Telefon fort
- **Ermöglicht freies Multitasking** — Starten Sie eine Generierung, durchsuchen Sie andere Tabs und kehren Sie zu einer vollständigen Antwort zurück

## Funktionsweise [#how-it-works]

Wenn Sie eine Nachricht an ein KI-Modell senden, erstellt LibreChat einen Generierungsauftrag, der alle gestreamten Inhalte nachverfolgt. Die Magie geschieht, wenn etwas Ihre Verbindung unterbricht:

1. **Automatische Erkennung** — Der Client erkennt die Trennung sofort
2. **Zustandswiederherstellung** — Nach dem erneuten Verbinden baut der Server alle zuvor gestreamten Inhalte wieder auf
3. **Nahtlose Synchronisierung** — Fehlende Inhalte werden über ein Synchronisierungsereignis bereitgestellt
4. **Transparente Fortsetzung** — Das Streaming wird ab der aktuellen Position fortgesetzt

Dies geschieht alles automatisch – es ist keine Benutzeraktion erforderlich.

### Multi-Tab- & Multi-Device-Erfahrung [#multi-tab--multi-device-experience]

Einer der leistungsfähigsten Aspekte von fortsetzbaren Streams ist die **Echtzeit-Synchronisierung**:

- **Derselbe Chat, mehrere Fenster** — Öffnen Sie eine Unterhaltung in zwei Browser-Tabs und beide erhalten gleichzeitig Aktualisierungen
- **Geräteübergreifende Kontinuität** — Starten Sie eine lange Generierung auf Ihrem Laptop und überprüfen Sie das Ergebnis anschließend auf Ihrem Smartphone
- **Team-Zusammenarbeit** — In geteilten Unterhaltungen sehen alle Betrachter, wie Inhalte in Echtzeit erscheinen

## Bereitstellungsmodi [#deployment-modes]

LibreChat unterstützt zwei Bereitstellungskonfigurationen:

### Single-Instance-Modus (Standard) [#single-instance-mode-default]

Verwendet In-Memory-Speicher mit Node.js `EventEmitter` für Pub/Sub. Perfekt für:
- Lokale Entwicklung
- Einzelserver-Bereitstellungen
- Docker Compose-Setups

**Keine Konfiguration erforderlich** — Funktioniert sofort nach der Installation.

### Redis-Modus (Produktion) [#redis-mode-production]

Verwendet Redis Streams und Pub/Sub für die Kommunikation zwischen Instanzen. Essenziell für:
- Horizontal skalierte Deployments
- Lastverteilte Produktionsumgebungen
- Hochverfügbarkeits-Setups
- Kubernetes-Cluster

Mit dem Redis-Modus kann ein Benutzer eine Generierung auf einer Serverinstanz starten und nahtlos auf einer anderen fortsetzen – perfekt für Rolling Deployments und Auto-Scaling.

**Hinweis:** Wenn Sie nur eine einzelne LibreChat-Instanz betreiben, ist Redis für fortsetzbare Streams in der Regel nicht erforderlich – der In-Memory-Modus erledigt alles. Redis wird wertvoll, wenn Sie mehrere LibreChat-Instanzen hinter einem Load Balancer haben. Dennoch ist Redis auch bei Bereitstellungen mit einer einzelnen Instanz für andere Funktionen wie Caching und Sitzungsspeicherung nützlich.

## Konfiguration [#configuration]

### Aktivieren von Redis Streams [#enabling-redis-streams]

Wenn Redis aktiviert ist (`USE_REDIS=true`), verwenden fortsetzbare Streams automatisch Redis. Sie können dies auch explizit aktivieren:

```bash filename=".env"
USE_REDIS=true
REDIS_URI=redis://localhost:6379
# Resumable streams will use Redis automatically when USE_REDIS=true
# To explicitly control it:
USE_REDIS_STREAMS=true
```

### Redis Cluster-Unterstützung [#redis-cluster-support]

Für Redis Cluster-Bereitstellungen:

```bash filename=".env"
USE_REDIS_STREAMS=true
USE_REDIS_CLUSTER=true
REDIS_URI=redis://node1:7001,redis://node2:7002,redis://node3:7003
```

LibreChat verwendet automatisch Hash-Tag-Schlüssel, um sicherzustellen, dass Multi-Key-Operationen innerhalb desselben Cluster-Slots bleiben.

## Anwendungsfälle [#use-cases]

### Instabile Netzwerke [#unstable-networks]
Bei instabilen WLAN- oder Mobilfunkverbindungen werden Antworten automatisch fortgesetzt, sobald die Verbindung wiederhergestellt ist. Sie müssen Ihren Prompt nicht erneut senden.

### Mobile-Benutzer [#mobile-users]
Wechseln Sie von WLAN zu Mobilfunk (oder umgekehrt), ohne Ihre Antwort zu verlieren. Der Stream wird genau dort fortgesetzt, wo er unterbrochen wurde.

### Lang andauernde Generierungen [#long-running-generations]
Bei komplexen Prompts, die lange Antworten generieren, können Sie gerne andere Tabs oder Apps aufrufen. Ihre Antwort wird auf Sie warten, wenn Sie zurückkehren.

### Multi-Device-Workflows [#multi-device-workflows]
Starten Sie eine Konversation auf Ihrem Arbeitscomputer, pendeln Sie nach Hause und überprüfen Sie das Ergebnis auf Ihrem Telefon – die vollständige Antwort ist dort verfügbar.

### Produktionsbereitstellungen [#production-deployments]
Skalieren Sie horizontal über mehrere Serverinstanzen hinweg und bewahren Sie dabei die Stream-Kontinuität. Rolling Deployments unterbrechen keine aktiven Generierungen.

## Technische Details [#technical-details]

### Inhaltsrekonstruktion [#content-reconstruction]

Das System aggregiert alle gestreamten Delta-Ereignisse, um Folgendes wiederherzustellen:
- Nachrichteninhalt (Text, Tool-Aufrufe, Zitate)
- Agent-Ausführungsschritte und Zwischenüberlegungen
- Metadaten und Statusinformationen

### Leistungsoptimierungen [#performance-optimizations]

**Memory-first-Ansatz**: Beim erneuten Verbinden mit derselben Serverinstanz verwendet LibreChat einen lokalen Cache für die Inhaltswiederherstellung ohne Latenz, wodurch unnötige Redis-Roundtrips vermieden werden.

**Automatische Bereinigung**: Veraltete Job-Einträge werden bei Abfragen entfernt, um Speicherlecks zu verhindern. Abgeschlossene Streams laufen automatisch ab.

**Effiziente Speicherung**: Der In-Memory-Modus verwendet `WeakRef` für die Graph-Speicherung, was eine automatische Garbage Collection ermöglicht, wenn Konversationen beendet werden.

### Datenfluss [#data-flow]

| Komponente | Speichermechanismus |
|-----------|-------------------|
| Chunks | Redis Streams (`XADD`/`XRANGE`) |
| Job-Metadaten | Redis Hash-Strukturen |
| Echtzeit-Ereignisse | Redis Pub/Sub-Kanäle |
| Ablauf | Automatisches TTL nach Abschluss des Streams |

## Testen von fortsetzbaren Streams [#testing-resumable-streams]

Sie können überprüfen, ob die Funktion funktioniert:

1. Starten Sie eine Streaming-Konversation mit einem beliebigen KI-Modell
2. **Tab-Test**: Öffnen Sie denselben Chat in einem neuen Browser-Tab – beide sollten synchronisiert werden
3. **Verbindungstest**: Schalten Sie Ihr Netzwerk kurz aus und verbinden Sie es dann wieder
4. **Navigationstest**: Navigieren Sie während des Streamings weg und kehren Sie dann zurück

In allen Fällen sollten Sie die vollständige Antwort ohne Datenverlust sehen.

## Fehlerbehebung [#troubleshooting]

### Streams werden nicht fortgesetzt? [#streams-not-resuming]

**Überprüfen der Redis-Konnektivität:**
```bash
docker exec -it librechat-redis redis-cli ping
# Should return: PONG
```

**Umgebungsvariablen überprüfen:**
```bash
# Ensure USE_REDIS_STREAMS is set
echo $USE_REDIS_STREAMS
```

### Inhalte erscheinen dupliziert? [#content-appears-duplicated]

Dies deutet normalerweise auf eine Diskrepanz der Client-Version hin. Stellen Sie sicher, dass Sie die neueste Version von LibreChat ausführen.

### Hohe Speicherauslastung im Einzelinstanz-Modus? [#high-memory-usage-in-single-instance-mode]

Abgeschlossene Streams werden automatisch durch Garbage Collection bereinigt. Wenn Sie eine hohe Speicherauslastung feststellen, prüfen Sie Folgendes:
- Sehr lange laufende Streams, die noch nicht abgeschlossen sind
- Streams, die ohne ordnungsgemäße Bereinigung fehlerhaft waren
