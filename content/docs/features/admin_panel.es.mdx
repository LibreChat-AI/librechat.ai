# Panel de administraciГіn de LibreChat [#librechat-admin-panel]

El **LibreChat Admin Panel** es una interfaz de gestiГіn independiente basada en navegador para LibreChat. Se conecta a la misma base de datos que LibreChat y proporciona una GUI para las tareas administrativas que potencian el [control de acceso granular](/docs/features/access_control): administraciГіn de usuarios y grupos, gestiГіn de roles, anulaciones de configuraciГіn con alcance a roles o grupos, y concesiones de capacidades a nivel de sistema.

<Callout type="info" title="Estado: Vista previa">
  El panel de administraciГіn ya estГЎ disponible para pruebas y es la prГіxima interfaz de gestiГіn que se basa en las APIs de administraciГіn introducidas en [LibreChat v0.8.5](/changelog/v0.8.5). El cГіdigo fuente, los problemas y las versiones se encuentran en [github.com/ClickHouse/librechat-admin-panel](https://github.com/ClickHouse/librechat-admin-panel).
</Callout>

## QuГ© hace [#what-it-does]

El panel de administraciГіn es un cliente ligero: todos los datos residen en la base de datos de LibreChat, y cada acciГіn pasa por los endpoints versionados `/api/admin/*` en el servidor de la API de LibreChat. Ofrece a los administradores un lugar Гәnico para:

- **Gestionar configuraciГіn**: vea y edite cada configuraciГіn de LibreChat a travГ©s de un formulario dinГЎmico basado en esquemas. Los nuevos campos aГұadidos al esquema de configuraciГіn aparecen automГЎticamente, sin necesidad de una versiГіn del panel de administraciГіn.
- **Aplicar anulaciones por principal**: limite las anulaciones de configuraciГіn a roles o grupos especГӯficos, con una cascada basada en prioridades que determina el valor final resuelto que cada usuario ve al iniciar sesiГіn.
- **Administrar usuarios**: listar, buscar y ver cada cuenta en la instancia.
- **Gestionar grupos**: crear y eliminar grupos, aГұadir/eliminar miembros y utilizar grupos como entidades de primer nivel en ACL y anulaciones.
- **Gestionar roles**: cree roles personalizados mГЎs allГЎ de los integrados `USER` / `ADMIN`, edite su matriz de permisos de funciones y asigne usuarios a los roles.
- **EmisiГіn de concesiones del sistema**: delegue capacidades de administraciГіn (por ejemplo, `manage:users`, `read:usage`, `manage:mcpservers`) a usuarios, grupos o roles especГӯficos sin convertirlos en administradores completos.
- **AutenticaciГіn**: inicie sesiГіn con una cuenta de administrador local de LibreChat, o mediante SSO de OpenID / SAML / proveedores de OAuth compatibles cuando estos estГ©n habilitados en la instancia de LibreChat.

Para conocer el modelo de permisos subyacente (principales, ACL de recursos, capacidades y cГіmo se componen las capas), consulte la pГЎgina [Access Control](/docs/features/access_control).

El panel de administraciГіn se ejecuta como un servicio independiente; no comparte proceso con LibreChat. Las capacidades de administraciГіn se verifican en el lado de LibreChat mediante la concesiГіn del sistema `access:admin` o el rol `SystemRoles.ADMIN`, por lo que el panel no puede otorgarse a sГӯ mismo privilegios que no deberГӯa tener.

La superficie de la API de administraciГіn expuesta por LibreChat es:

| Montaje | PropГіsito |
| ----------------------------------------- | --------------------------------------------------------- |
| `POST /api/admin/login` &nbsp; `/oauth/*` | Endpoints de autenticaciГіn especГӯficos para administradores (local + SSO) |
| `GET /api/admin/verify`                   | Valida la sesiГіn de administrador |
| `/api/admin/users`                        | Listado y bГәsqueda de usuarios |
| `/api/admin/groups`                       | CRUD de grupos + gestiГіn de miembros |
| `/api/admin/roles`                        | CRUD de roles personalizados + ediciГіn de permisos + gestiГіn de miembros |
| `/api/admin/grants`                       | Concesiones de capacidades del sistema (asignar/revocar/listar) |
| `/api/admin/config`                       | ConfiguraciГіn base + anulaciones por principal |

## Primeros pasos [#getting-started]

### Requisitos previos [#prerequisites]

- Una instancia de LibreChat en ejecuciГіn en la **v0.8.5 o posterior** (las APIs de administraciГіn no estГЎn disponibles en versiones anteriores)
- Acceso de red desde el contenedor/host del admin-panel a la API de LibreChat
- Una cuenta de administrador en LibreChat: ya sea el primer usuario registrado (auto-admin), un usuario con `role: 'ADMIN'` configurado en Mongo, o un principal al que se le haya otorgado la capacidad `access:admin`

### Incluido con LibreChat (recomendado) [#bundled-with-librechat-recommended]

Si ejecutas LibreChat con su [`docker-compose.yml`](https://github.com/danny-avila/LibreChat/blob/main/docker-compose.yml) o [`deploy-compose.yml`](https://github.com/danny-avila/LibreChat/blob/main/deploy-compose.yml) oficial, el panel de administraciГіn se incluye como un servicio y se inicia automГЎticamente junto con LibreChat; no se requiere una implementaciГіn por separado.

| Archivo Compose | URL del panel de administraciГіn | CГіmo se sirve |
| ------------------------------ | ------------------------ | ----------------------------------------------------------------- |
| `docker-compose.yml` (predeterminado) | `http://localhost:3000`  | Publicado en un puerto del host (`ADMIN_PANEL_PORT`, predeterminado `3000`)     |
| `deploy-compose.yml`           | `http://admin.localhost` | Enrutado a travГ©s del proxy inverso nginx incluido en un subdominio     |

Establezca el secreto de sesiГіn del panel en el archivo `.env` de LibreChat antes de iniciar la pila; los archivos compose lo pasan como el `SESSION_SECRET` del panel:

```bash filename=".env"
# Min 32 characters. Generate with: openssl rand -hex 32
ADMIN_PANEL_SESSION_SECRET=replace-with-a-32-char-random-string

# Optional: host port for the default docker-compose
# ADMIN_PANEL_PORT=3000

# Optional: set true when the panel is served over HTTPS
# ADMIN_PANEL_SESSION_COOKIE_SECURE=false
```

Los archivos compose conectan el resto automГЎticamente: `API_SERVER_URL` apunta al servicio `api`, `VITE_API_BASE_URL` sigue a `DOMAIN_CLIENT` para las redirecciones OAuth orientadas al navegador, y `ADMIN_PANEL_URL` se configura para que LibreChat devuelva a los administradores al panel despuГ©s del SSO. Para optar por no participar, elimine el servicio `admin-panel` o protГ©jalo detrГЎs de una entrada de [`profiles`](https://docs.docker.com/compose/how-tos/profiles/) de Compose.

<Callout type="info" title="admin.localhost en un dominio real">
  Los navegadores modernos resuelven `*.localhost` (incluyendo `admin.localhost`) a `127.0.0.1`, por lo que la URL de deploy-compose funciona sin cambios en el archivo hosts. Para un dominio real, apunte un registro DNS al host, actualice el `server_name` de `admin.localhost` en `client/nginx.conf` y configure `ADMIN_PANEL_URL` para que coincida.
</Callout>

### Standalone (despliegue independiente) [#standalone-separate-deployment]

Para alojar el panel de administraciГіn por su cuenta (apuntando a una instancia de LibreChat que se ejecuta en otro lugar), utilice la imagen publicada desde GHCR:

```bash
# 1. Create an env file
cp .env.example .env

# 2. Edit .env and set at minimum:
#    SESSION_SECRET=<random string, min 32 characters>
#    VITE_API_BASE_URL=http://host.docker.internal:3080

# 3. Start it
docker compose up -d   # http://localhost:3000
docker compose down    # stop
```

`docker run` independiente:

```bash
docker run -p 3000:3000 \
  --add-host=host.docker.internal:host-gateway \
  -e SESSION_SECRET=replace-with-32-char-random-string \
  -e VITE_API_BASE_URL=http://host.docker.internal:3080 \
  ghcr.io/clickhouse/librechat-admin-panel:latest
```

<Callout type="warning" title="Redes de Docker">
  Dentro de un contenedor, `localhost` se refiere al contenedor mismo, no a su host. Cuando LibreChat se ejecuta en el mismo host, apunte `VITE_API_BASE_URL` a `http://host.docker.internal:3080` (Linux: aГұada `--add-host=host.docker.internal:host-gateway`). En producciГіn, utilice el nombre DNS pГәblico/interno de su API de LibreChat.
</Callout>

### Ejecutar localmente para desarrollo [#run-locally-for-development]

```bash
git clone https://github.com/ClickHouse/librechat-admin-panel.git
cd librechat-admin-panel
cp .env.example .env    # then edit

Гҡtil cuando el servidor del panel de administraciГіn accede a LibreChat a travГ©s de una URL diferente a la del navegador (ej. servicio interno de Kubernetes frente a nombre de host pГәblico). |
| `PORT` | No | `3000` | Puerto en el que escucha el panel de administraciГіn. |
| `ADMIN_PANEL_SESSION_SECRET` | **SГӯ** en producciГіn | Utiliza `CREDS_KEY` como alternativa en los stacks de Docker de LibreChat incluidos | Variable del lado de LibreChat asignada al `SESSION_SECRET` del panel de administraciГіn para el servicio del panel de administraciГіn incluido. | `ADMIN_PANEL_PORT` | No | `3000` | Puerto del host expuesto por el servicio del panel de administraciГіn incluido en el `docker-compose.yml` predeterminado. |
| `ADMIN_SSO_ONLY` | No | `false` | Oculta el formulario de correo electrГіnico/contraseГұa, forzando el inicio de sesiГіn solo mediante SSO. |

En las pilas de Docker incluidas en LibreChat, el panel de administraciГіn se ejecuta como un servicio `admin-panel`. El `docker-compose.yml` predeterminado lo expone en `ADMIN_PANEL_PORT`; `deploy-compose.yml` lo enruta a travГ©s de nginx en `http://admin.localhost` y establece `ADMIN_PANEL_URL` para el servicio de API.

### URL de redirecciГіn de LibreChat [#librechat-redirect-url]

Cuando el panel de administraciГіn se aloja en una URL separada de LibreChat, configure `ADMIN_PANEL_URL` en el entorno de la API de LibreChat. Utilice la URL base externa del panel de administraciГіn, incluyendo cualquier prefijo de ruta, y omita la barra diagonal final:

```bash filename=".env"
ADMIN_PANEL_URL=https://admin.example.com/admin
```

Para despliegues con Helm, configure `librechat.adminPanelUrl` en su archivo values. El chart lo renderiza como `ADMIN_PANEL_URL` para el flujo de OAuth de administraciГіn de LibreChat:

```yaml filename="values.yaml"
librechat:
  adminPanelUrl: https://admin.example.com/admin
```

Para OpenID SSO, registre `${DOMAIN_SERVER}/api/admin/oauth/openid/callback` con su proveedor de identidad.

### Controles de cachГ© [#cache-controls]

Estos reflejan las variables de entorno de cachГ© de LibreChat. Las variantes `ADMIN_PANEL_*` tienen prioridad, recurriendo a los equivalentes compartidos de LibreChat cuando no estГЎn configuradas.

| Variable | PropГіsito |
| --------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `STATIC_CACHE_MAX_AGE` / `ADMIN_PANEL_STATIC_CACHE_MAX_AGE`     | `max-age` del navegador en segundos para los activos con hash en `/assets/` (predeterminado 172800 = 2 dГӯas). |
| `STATIC_CACHE_S_MAX_AGE` / `ADMIN_PANEL_STATIC_CACHE_S_MAX_AGE` | `s-maxage` de CDN en segundos (predeterminado 86400 = 1 dГӯa). |
| `INDEX_CACHE_CONTROL` / `ADMIN_PANEL_INDEX_CACHE_CONTROL`       | Encabezado `Cache-Control` para la respuesta del Гӯndice HTML. |
| `INDEX_PRAGMA` / `ADMIN_PANEL_INDEX_PRAGMA`                     | Encabezado `Pragma` para la respuesta del Гӯndice HTML. |
| `INDEX_EXPIRES` / `ADMIN_PANEL_INDEX_EXPIRES`                   | Encabezado `Expires` para la respuesta del Гӯndice HTML. |

## AutenticaciГіn [#authentication]

El panel de administraciГіn reutiliza la pila de autenticaciГіn de LibreChat y no tiene su propia base de datos de usuarios. Se admiten dos rutas de inicio de sesiГіn:

- **Cuentas locales**: nombre de usuario/contraseГұa para cualquier usuario de LibreChat cuya cuenta supere la verificaciГіn de acceso de administrador.
- **Single sign-on**: OpenID Connect, SAML y los proveedores de OAuth sociales ya configurados en su instancia de LibreChat. Establezca `ADMIN_SSO_ONLY=true` para ocultar el formulario de contraseГұa por completo.

El acceso de administrador es verificado en el lado del servidor por LibreChat para cada solicitud. La cuenta debe cumplir con una de las siguientes condiciones:

1. Tener `role: 'ADMIN'` en MongoDB, **o**
2. Tener el permiso de sistema `access:admin` (asignado a otro principal a travГ©s del propio panel de administraciГіn; consulte [System Grants](/docs/features/access_control#layer-3-system-grants-admin-capabilities)).

Las sesiones estГЎn basadas en cookies, cifradas con `SESSION_SECRET` y expiran por inactividad segГәn `ADMIN_SESSION_IDLE_TIMEOUT_MS`.

## GestiГіn de configuraciГіn [#configuration-management]

El panel renderiza la configuraciГіn de LibreChat como un formulario dinГЎmico basado en el esquema de configuraciГіn. Esto tiene dos propiedades Гәtiles:

- **Compatible con versiones futuras**: cuando LibreChat lanza un nuevo campo de configuraciГіn, el panel lo detecta automГЎticamente a partir del esquema. No se requiere ninguna actualizaciГіn ni redistribuciГіn del panel de administraciГіn.
- **Anulaciones en capas**: la configuraciГіn base (desde `librechat.yaml`) puede ser ensombrecida por anulaciones por principal delimitadas a un rol o grupo. Cuando un usuario inicia sesiГіn, las anulaciones se resuelven en orden de prioridad y se fusionan sobre la base para producir la configuraciГіn efectiva que el usuario ve.

Esta es la superficie detrГЎs del [sistema de anulaciГіn de configuraciГіn por principal respaldado por BD](https://github.com/danny-avila/LibreChat/pull/12354) de LibreChat. Casos de uso tГӯpicos:

- Asignar a un grupo "Research" un `recursionLimit` mГЎs alto y endpoints adicionales
- Permitir que un rol "FinanceAdmins" gestione servidores MCP mientras que los usuarios regulares solo puedan usarlos
- Limitar los permisos de `interface` mГЎs estrictos a los grupos de contratistas externos
