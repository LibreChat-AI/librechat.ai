Dans Docker Compose, un fichier de remplacement (override file) est une fonctionnalité puissante qui vous permet de modifier la configuration par défaut fournie par le `docker-compose.yml` principal sans avoir besoin de modifier directement ou de dupliquer le fichier entier. L'utilisation principale du fichier de remplacement concerne les personnalisations pour le développement local, et Docker Compose fusionne les configurations des fichiers `docker-compose.yml` et `docker-compose.override.yml` lorsque vous exécutez `docker compose up`.

Voici un guide rapide sur la façon d'utiliser le `docker-compose.override.yml` :

> Remarque : Veuillez consulter le fichier `docker-compose.override.yml.example` pour plus d'exemples

Consultez la documentation officielle de Docker pour plus d'informations :

- **[docker docs - understanding-multiple-compose-files](https://docs.docker.com/compose/multiple-compose-files/extends/#understanding-multiple-compose-files)**
- **[docker docs - merge-compose-files](https://docs.docker.com/compose/multiple-compose-files/merge/#merge-compose-files)**
- **[docker docs - spécification-de-fichiers-compose-multiples](https://docs.docker.com/compose/reference/#specifying-multiple-compose-files)**

## Étape 1 : Créer un fichier `docker-compose.override.yml` [#step-1-create-a-docker-composeoverrideyml-file]

Si vous n'avez pas encore de fichier `docker-compose.override.yml`, vous pouvez en créer un en copiant le contenu de l'exemple de remplacement :

```bash filename="Copying the example override file"
cp docker-compose.override.yml.example docker-compose.override.yml
```

Ce fichier sera automatiquement pris en compte par Docker Compose lorsque vous exécuterez des commandes docker-compose.

## Étape 2 : Modifier le fichier de remplacement [#step-2-edit-the-override-file]

Ouvrez votre fichier `docker-compose.override.yml` avec vscode ou n'importe quel éditeur de texte.

Apportez les modifications souhaitées en décommentant les sections pertinentes et en les personnalisant selon vos besoins.

> Avertissement : Vous ne pouvez spécifier chaque nom de service qu'une seule fois (api, mongodb, meilisearch, ...). Si vous souhaitez remplacer plusieurs paramètres dans un même service, vous devrez effectuer les modifications en conséquence.

### Exemples [#examples]

Si vous voulez vous assurer que Docker puisse utiliser votre fichier `librechat.yaml` pour les [Custom Endpoints & Configuration](/docs/configuration/librechat_yaml), cela ressemblerait à ceci :

```yaml filename="docker-compose.override.yml"
services:
  api:
    volumes:
      - ./librechat.yaml:/app/librechat.yaml
```

Ou, si vous souhaitez construire localement l'image pour le service `api`, utiliser le fichier de configuration LibreChat, et utiliser l'ancienne version de Mongo qui ne nécessite pas la prise en charge d'AVX, votre `docker-compose.override.yml` pourrait ressembler à ceci :

```yaml filename="docker-compose.override.yml"
services:
  api:
    volumes:
      - ./librechat.yaml:/app/librechat.yaml
    image: librechat
    build:
      context: .
      target: node

  mongodb:
    image: mongo:4.4.18
```

> Remarque : Soyez prudent si vous exposez des ports pour MongoDB ou Meilisearch au public, car cela peut rendre vos données vulnérables.

## Étape 3 : Appliquer les changements [#step-3-apply-the-changes]

Pour appliquer vos modifications de configuration, exécutez simplement Docker Compose comme d'habitude. Docker Compose prend automatiquement en compte à la fois les fichiers `docker-compose.yml` et `docker-compose.override.yml` :

```bash filename="Apply the changes"
docker compose up -d
```

## Étape 4 : Vérifier les changements [#step-4-verify-the-changes]

Après avoir démarré vos services avec la configuration modifiée, vous pouvez vérifier que les changements ont été appliqués en utilisant la commande `docker ps` pour lister les conteneurs en cours d'exécution et leurs propriétés, telles que les ports.

## Considérations importantes [#important-considerations]

- **Ordre de priorité** : Les valeurs définies dans le fichier de remplacement (override) ont priorité sur celles spécifiées dans le fichier `docker-compose.yml` original.
- **Sécurité** : Lors de la personnalisation des ports et de l'exposition publique de services, soyez toujours conscient des implications en matière de sécurité. Évitez d'utiliser les valeurs par défaut pour la production ou les environnements sensibles.

En suivant ces étapes et considérations, vous pouvez facilement et en toute sécurité modifier votre configuration Docker Compose sans altérer le fichier `docker-compose.yml` original, ce qui simplifie la gestion et la maintenance de différents environnements ou personnalisations locales.


## `deploy-compose.yml` [#deploy-composeyml]

Pour utiliser un fichier de remplacement avec un fichier Docker Compose non par défaut, tel que `deploy-compose.yml`, vous devrez spécifier explicitement les deux fichiers lors de l'exécution des commandes Docker Compose.

Docker Compose vous permet de spécifier plusieurs options `-f` ou `--file` pour inclure plusieurs fichiers compose, où les paramètres des fichiers suivants remplacent ou complètent ceux du premier.

Si vous utilisez `deploy-compose.yml` comme configuration principale Docker Compose et que vous avez un fichier de remplacement nommé `docker-compose.override.yml` (vous pouvez nommer le fichier de remplacement comme vous le souhaitez, mais il est possible que vous ayez déjà ce fichier spécifique), vous exécuteriez les commandes Docker Compose comme suit :

```bash
docker compose -f deploy-compose.yml -f docker-compose.override.yml pull
docker compose -f deploy-compose.yml -f docker-compose.override.yml up
```
