## fileStrategy

- **Opciones**: "local" | "firebase" | "s3" | "azure_blob" | "cloudfront"

<OptionTable
  options={[
    [
      'fileStrategy',
      'String',
      'Determina dónde guardar los archivos subidos o generados por el usuario. El valor predeterminado es `"local"` si se omite.',
      'fileStrategy: "firebase"',
    ],
  ]}
/>

- **Notas**:
  - `"cloudfront"` almacena archivos en S3 y devuelve URLs de CloudFront para una entrega de medios estable, cookies firmadas y descargas firmadas.
  - `"firebase"` sirve archivos a través de Firebase Storage y las ubicaciones perimetrales (edge locations) de Firebase Hosting.
  - S3 sirve archivos a través de **presigned URLs** (tokens firmados temporales) que caducan. Una vez caducados, cualquier imagen o avatar que haga referencia a esa URL aparecerá roto en la UI. Esto hace que S3 no sea adecuado como estrategia principal para activos visuales. Consulta la [related discussion](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903) para más detalles.
  - Para obtener el mejor rendimiento de imágenes y avatares, utilice `"cloudfront"` o `"firebase"`, o configure `fileStrategies` para dirigir `avatar` e `image` a una estrategia respaldada por una CDN.
  - Consulte la [documentación de File Storage & CDN](/docs/configuration/cdn) para obtener detalles sobre la configuración.

## fileStrategies

Permite un control granular sobre las estrategias de almacenamiento de archivos para diferentes tipos de archivos.

- **Estrategias disponibles**: "local" | "firebase" | "s3" | "azure_blob" | "cloudfront"

<OptionTable
  options={[
    [
      'fileStrategies',
      'Object',
      'Configura diferentes estrategias de almacenamiento para distintos tipos de archivos. Más flexible que la opción de fileStrategy única.',
      '',
    ],
  ]}
/>

**Sub-claves:**

<OptionTable
  options={[
    [
      'default',
      'String',
      'Estrategia de almacenamiento de respaldo cuando no se define un tipo específico. El valor predeterminado es "local".',
      '',
    ],
    [
      'avatar',
      'String',
      'Estrategia de almacenamiento para imágenes de avatar de usuario y agente. Se recomienda utilizar una estrategia respaldada por CDN (`"cloudfront"` o `"firebase"`) para obtener el mejor rendimiento.',
      '',
    ],
    [
      'image',
      'String',
      'Estrategia de almacenamiento para imágenes subidas en los chats. Se recomienda utilizar una estrategia respaldada por CDN (`"cloudfront"` o `"firebase"`) para obtener el mejor rendimiento.',
      '',
    ],
    ['document', 'String', 'Estrategia de almacenamiento para la carga de documentos (PDFs, archivos de texto, etc.).', ''],
    ['skills', 'String', 'Estrategia de almacenamiento para archivos incluidos con Skills.', ''],
  ]}
/>

- **Notas**:
  - Esta configuración tiene prioridad sobre la opción única `fileStrategy`
  - Si un tipo de archivo específico no está configurado, recurre a `default`, luego a `fileStrategy` y finalmente a `"local"`.
  - Las imágenes y los avatares necesitan URLs persistentes y estables para renderizarse correctamente en toda la interfaz de usuario. Las URLs prefirmadas de S3 caducan (límite de AWS: 7 días para usuarios IAM, horas para credenciales basadas en roles/STS), lo que provoca que las imágenes se rompan en el selector de modelos y en la interfaz de chat. Consulta la [discusión relacionada](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903) para obtener el contexto completo. Utiliza `"cloudfront"` o `"firebase"` para `avatar` e `image` para evitar esto.
  - S3 y Azure Blob Storage son adecuados para el almacenamiento de `document`, donde las URLs de descarga prefirmadas de corta duración son apropiadas.
  - Consulte la [documentación de File Storage & CDN](/docs/configuration/cdn) para obtener detalles de configuración para cada proveedor de almacenamiento.

**Ejemplos:**

```yaml filename="fileStrategies - All in one place"
# Use a single strategy for all file types
fileStrategies:
  default: 's3'
```

```yaml filename="fileStrategies - Mixed strategies"
# Route images and avatars to CDN, keep documents in object storage
fileStrategies:
  avatar: 'cloudfront' # CDN delivery for avatars
  image: 'cloudfront' # CDN delivery for generated/uploaded images
  document: 's3' # Object storage for documents
```

```yaml filename="fileStrategies - Partial configuration"
# Only configure specific types, others use default
fileStrategies:
  default: 'local'
  avatar: 'firebase' # Only avatars use Firebase CDN, everything else is local
```

## clientImageResize

**Clave:**

<OptionTable
  options={[
    [
      'clientImageResize',
      'Object',
      'Configura el cambio de tamaño de imágenes en el lado del cliente para optimizar las cargas de archivos y evitar errores de carga debido a tamaños de imagen grandes.',

**Subclaves:**

<OptionTable
  options={[
    [
      'enabled',
      'Boolean',
      'Habilita o deshabilita la funcionalidad de cambio de tamaño de imágenes del lado del cliente. Predeterminado: false.',
      'enabled: true',
    ],

      'maxWidth',
      'Number',
      'Ancho máximo en píxeles para imágenes redimensionadas. Las imágenes más anchas que esto serán redimensionadas. Predeterminado: 1920.',
      'maxWidth: 1024',
    ],
    [
      'maxHeight',
      'Number',
      'Altura máxima en píxeles para imágenes redimensionadas. Las imágenes más altas que esto serán redimensionadas. Predeterminado: 1080.',
      'maxHeight: 768',
    ],
    [
      'quality',
      'Number',
      'Calidad de compresión JPEG (0.1 a 1.0). Los valores más altos significan mejor calidad pero tamaños de archivo más grandes. Predeterminado: 0.8.',
      'quality: 0.9',
    ],
    [
      'compressFormat',
      'String',
      'Formato de salida para imágenes comprimidas. Opciones: "jpeg", "webp". Predeterminado: "jpeg".',
      'compressFormat: "webp"',
    ],
  ]}
/>

**Descripción:**

La configuración `clientImageResize` permite el cambio de tamaño automático de imágenes en el lado del cliente antes de la carga. Esta función ayuda a:

- **Evitar errores de carga** debido a archivos de imagen grandes que exceden los límites del servidor
- **Reduce el uso de ancho de banda** comprimiendo las imágenes antes de su transmisión
- **Mejora el rendimiento de carga** con tamaños de archivo más pequeños
- **Mantener la calidad de la imagen** mientras se optimiza el tamaño del archivo

Cuando está habilitado, las imágenes que exceden las dimensiones `maxWidth` o `maxHeight` especificadas se redimensionan automáticamente en el lado del cliente antes de ser cargadas al servidor. El redimensionamiento mantiene la relación de aspecto original mientras asegura que la imagen se ajuste dentro de los límites especificados.

**Ejemplo:**

```yaml filename="clientImageResize"
fileConfig:
  clientImageResize:
    enabled: true
    maxWidth: 1920
    maxHeight: 1080
    quality: 0.8
    compressFormat: 'jpeg'
```

**Notas:**

- Solo se aplica a archivos de imagen (JPEG, PNG, WebP, etc.)
- El cambio de tamaño ocurre automáticamente cuando las imágenes exceden las dimensiones especificadas
- La relación de aspecto original se conserva durante el cambio de tamaño
- La función funciona con todos los endpoints de carga de imágenes compatibles
- El ajuste de calidad solo se aplica a los formatos JPEG y WebP
- Establecer una calidad demasiado baja (< 0.5) puede resultar en una degradación notable de la imagen

## mcpSettings

**Clave:**

<OptionTable
  options={[
    [
      'mcpSettings',
      'Object',
      'Define los ajustes globales para los servidores de Model Context Protocol (MCP)',
      '',
    ],
  ]}
/>

**Subclaves:**

<OptionTable
  options={[
    [
      'allowedDomains',
      'Array of Strings',
      'Lista blanca estricta de dominios para conexiones de servidores MCP. Cuando se establece, solo se puede acceder a las entradas listadas.',
      '',
    ],
    [
      'allowedAddresses',
      'Array of Strings',
      'Lista de exención de SSRF (solo espacio IP privado). Permite servicios específicos de host:puerto privados sin cambiar `allowedDomains` al modo de lista blanca estricta.',
      '',
    ],
  ]}
/>

- **Notas**:
  - Esta es una característica de seguridad para proteger contra el abuso / uso indebido de direcciones internas a través de servidores MCP
  - De forma predeterminada, LibreChat restringe que los servidores MCP se conecten a direcciones de red internas, locales o privadas.
  - Los servidores MCP que utilizan direcciones IP locales o dominios pueden añadirse a la lista blanca estricta `allowedDomains` (que entonces se convierte en el único conjunto accesible), o bien, para mantener accesibles los destinos públicos, pueden excluirse como servicios exactos de host:puerto a través de `allowedAddresses`
  - Como ocurre con todos los cambios de configuración en yaml, es necesario reiniciar LibreChat para que surtan efecto.
  - Admite dominios, subdominios comodín (`*.example.com`), dominios de Docker y direcciones IP

**Ejemplo:**

```yaml filename="mcpSettings"
mcpSettings:
  # Strict whitelist mode:
  # allowedDomains:
  #   - "example.com"           # Specific domain
  #   - "*.example.com"         # All subdomains
  #   - "http://mcp-server:3000" # Internal service, explicitly whitelisted

  # Default SSRF mode with private service exemptions:
  allowedAddresses:
    - 'host.docker.internal:8080' # Permit one private host on one port
    - '10.0.0.5:8000' # Permit one private IP on one port
```

## mcpServers

**Clave:**

<OptionTable
  options={[
    [
      'mcpServers',
      'Object',
      'Define la configuración para los servidores del Model Context Protocol (MCP), permitiendo la integración dinámica de servidores MCP dentro de la aplicación.',
      '',
    ],
  ]}
/>

**Subclaves:**

<OptionTable
  options={[
    [
      '<serverName>',
      'Object',
      'Cada clave bajo `mcpServers` representa una configuración individual de servidor MCP, identificada por un nombre único.',
      '',
    ],
  ]}
/>

- **Notas**:
  - La inicialización ocurre al iniciar, y la aplicación debe reiniciarse para que los cambios surtan efecto.
  - El `<serverName>` es un identificador único para cada configuración de servidor MCP.
  - Cada servidor MCP puede configurarse utilizando uno de los cuatro tipos de conexión:
    - `stdio`
    - `websocket`
    - `sse`
    - `streamable-http`
  - El campo `type` especifica el tipo de conexión al servidor MCP.
  - Si se omite `type`, se establece de forma predeterminada según la presencia y el formato de `url` o `command`:
    - Si se especifica `url` y comienza con `http` o `https`, `type` se establece de forma predeterminada en `sse`.
    - Si se especifica `url` y comienza con `ws` o `wss`, `type` toma `websocket` como valor predeterminado.
    - Si se especifica `command`, `type` toma `stdio` como valor predeterminado.
  - Las opciones de configuración adicionales incluyen:
    - `timeout`: Tiempo de espera en milisegundos para las solicitudes al servidor MCP. Determina cuánto tiempo esperar una respuesta para las solicitudes de herramientas.
    - `initTimeout`: Tiempo de espera en milisegundos para la inicialización del servidor MCP. Determina cuánto tiempo esperar a que el servidor se inicialice.
    - `serverInstructions`: Controla si las instrucciones del servidor se incluyen en el contexto del agente. Puede ser `true` (usar las proporcionadas por el servidor), `false` (desactivado) o una cadena personalizada (anula las proporcionadas por el servidor).
    - `customUserVars`: (Opcional) Define variables personalizadas (por ejemplo, claves de API, URLs) que los usuarios individuales pueden configurar para un servidor MCP. Estos valores por usuario, proporcionados a través de la interfaz de usuario, pueden ser referenciados en las configuraciones de `headers` o `env` del servidor utilizando la sintaxis `{{VARIABLE_NAME}}`. Esto permite la autenticación o personalización por usuario para las herramientas MCP.
  - ver: [Estructura del objeto MCP Servers](/docs/configuration/librechat_yaml/object_structure/mcp_servers)

**Ejemplo:**

```yaml filename="mcpServers"
mcpServers:
  everything:
    # type: sse # type can optionally be omitted
    url: http://localhost:3001/sse
    timeout: 30000
    initTimeout: 10000
    serverInstructions: true # Use server-provided instructions
  puppeteer:
    type: stdio
    command: npx
    args:
      - -y
      - '@modelcontextprotocol/server-puppeteer'
    timeout: 30000
    initTimeout: 10000
    serverInstructions: 'Do not access any local files or local/internal IP addresses'
  filesystem:
    # type: stdio
    command: npx
    args:
      - -y
      - '@modelcontextprotocol/server-filesystem'
      - /home/user/LibreChat/
    iconPath: /home/user/LibreChat/client/public/assets/logo.svg
  mcp-obsidian:
    command: npx
    args:
      - -y
      - 'mcp-obsidian'
      - /path/to/obsidian/vault
  streamable-http-example:
    type: streamable-http
    url: https://example.com/mcp
    headers:
      Authorization: 'Bearer ${API_TOKEN}'
    timeout: 30000
  per-user-crendentials-example:
    type: sse
    url: 'https//some.mcp/sse'
    headers:
      X-Custom-Auth-Token: '{{USER_API_KEY}}' # Placeholder for the user-provided API key, defined in `customUserVars` below.
    customUserVars:
      USER_API_KEY:
        title: 'Service API Key'
        description: "Your personal API key for this service. You can get it <a href='https://example.com/api-keys' target='_blank'>here</a>."
    serverInstructions: true
```
