## Übersicht [#overview]

Das `interface` Objekt ermöglicht die Anpassung verschiedener Benutzeroberflächenelemente innerhalb der Anwendung, einschließlich Sichtbarkeits- und Verhaltenseinstellungen für Komponenten wie Menüs, Panels und Links. Dieser Abschnitt bietet eine detaillierte Aufschlüsselung der Struktur des `interface` Objekts.

Dies sind Felder unter `interface`:

- `mcpServers`
- `privacyPolicy`
- `termsOfService`
- `modelSelect`
- `parameters`
- `contextUsage`
- `contextCost`
- `currency`
- `presets`
- `prompts`
- `bookmarks`
- `memories`
- `multiConvo`
- `agents`
- `remoteAgents`
- `skills`
- `sharedLinks`
- `temporaryChat`
- `temporaryChatRetention`
- `retentionMode`
- `retainAgentFiles`
- `autoSubmitFromUrl`
- `customWelcome`
- `runCode`
- `webSearch`
- `fileSearch`
- `fileCitations`
- `defaultPinnedTools`
- `peoplePicker`
- `marketplace`

**Hinweise:**

- Die `interface`-Konfigurationen werden global innerhalb der Anwendung angewendet.
- Für die meisten Einstellungen sind Standardwerte vorgesehen, diese können jedoch basierend auf spezifischen Anforderungen oder Bedingungen überschrieben werden.
- Die bedingte Logik in der Anwendung kann diese Einstellungen basierend auf anderen Konfigurationen, wie z. B. Model Specs, weiter modifizieren.

<Callout type="warning" title="Veraltet: Felder für Berechtigungs-Seiteneffekte">
Mehrere der unten aufgeführten Felder (`mcpServers`, `prompts`, `bookmarks`, `memories`, `multiConvo`, `agents`, `remoteAgents`, `skills`, `sharedLinks`, `temporaryChat`, `runCode`, `webSearch`, `fileSearch`, `fileCitations`, `peoplePicker`, `marketplace`) schalten nicht nur die Benutzeroberfläche um, sondern legen beim Start Rollenberechtigungen in der Datenbank fest, und zwar ausschließlich für die integrierte `USER`-Rolle.

Für die laufende Verwaltung verwenden Sie das [**LibreChat Admin Panel**](/docs/features/admin_panel), das die Berechtigungsmatrix direkt für jede Rolle (einschließlich benutzerdefinierter Rollen) bearbeitet. Diese YAML-Felder werden weiterhin für das Bootstrapping einer neuen Instanz oder für vollständig dateibasierte Bereitstellungen unterstützt, sollten jedoch nicht mehr als primäre Methode zur Verwaltung von Funktionsberechtigungen verwendet werden.

Siehe [Access Control](/docs/features/access_control) für das vollständige Berechtigungsmodell.

</Callout>

```yaml filename="interface"
    modalContent: |
      # Terms of Service

      Welcome to LibreChat!
  modelSelect: false
  parameters: true
  contextUsage: true
  contextCost: true
  currency:
    code: EUR
    rate: 0.92
  presets: false
  prompts:
    use: true
    create: true
    share: true
    public: false
  bookmarks: true

    use: true
    create: true

    - artifacts
    - execute_code
    - mcp
```

## prompts

> **Veraltet für die Berechtigungsverwaltung.** Initialisiert die `PROMPTS`-Rollenberechtigungen beim Start nur für die Standard-`USER`-Rolle. Verwenden Sie bevorzugt das [Admin Panel](/docs/features/admin_panel), um Prompt-Berechtigungen pro Rolle/Gruppe/Benutzer zu verwalten.

**Schlüssel:**

<OptionTable
  options={[
    [
      'prompts',
      'Boolean or Object',
      'Steuert Prompt-bezogene Funktionen für alle Benutzer. Kann ein Boolean für einfaches Aktivieren/Deaktivieren sein oder ein Objekt für eine granulare Steuerung von Nutzung, Erstellung, Teilen und öffentlicher Sichtbarkeit.',
      'When set to `false`, users will not have access to create, edit, or use custom prompts.',
    ],
  ]}
/>

**Standard:** `true`

**Wichtig: Boolean vs. Objekt-Konfiguration**

- **Boolean (`prompts: true`)**: Aktualisiert nur die `use`-Berechtigung. Bestehende `create`-, `share`- und `public`-Berechtigungswerte werden aus der Datenbank **beibehalten**. Verwenden Sie dies als einfachen Funktionsschalter, ohne andere Einstellungen zu beeinflussen, die über das Admin-Panel konfiguriert wurden.

- **Objekt**: Aktualisiert nur die Unterberechtigungen, die explizit angegeben sind. Alle Berechtigungen, die nicht in der Konfiguration enthalten sind, werden aus der Datenbank beibehalten.

Bei Verwendung der Objektstruktur:

## temporaryChatRetention

Die `temporaryChatRetention`-Konfiguration ermöglicht es Ihnen anzupassen, wie lange temporäre Chats gespeichert werden, bevor sie automatisch gelöscht werden.

**Schlüssel:**

<OptionTable
  options={[
    [
      'temporaryChatRetention',
      'Number',
      'Legt den Aufbewahrungszeitraum für temporäre Chats in Stunden fest.',
      'temporaryChatRetention: 168',
    ],
  ]}
/>

**Validierungsregeln:**

- **Minimum**: 1 Stunde (verhindert sofortiges Löschen)
- **Maximum**: 8760 Stunden (maximal 1 Jahr Aufbewahrungsdauer)
- **Standard**: 720 Stunden (30 Tage)

**Konfigurationsmethoden:**

1. **LibreChat.yaml** (empfohlen): `interface.temporaryChatRetention: 168`
2. **Umgebungsvariable** (veraltet): `TEMP_CHAT_RETENTION_HOURS=168`

> **Hinweis:** Die Umgebungsvariable `TEMP_CHAT_RETENTION_HOURS` ist veraltet. Bitte verwenden Sie stattdessen die Konfigurationsoption `interface.temporaryChatRetention` in `librechat.yaml`. Der Wert in der Konfigurationsdatei hat Vorrang vor der Umgebungsvariablen.

**Beispiel:**

```yaml filename="interface / temporaryChatRetention"
interface:
  temporaryChatRetention: 168 # Retain temporary chats for 7 days
  retentionMode: 'temporary'
```

**Gängige Aufbewahrungsfristen:**

- **1 Stunde**: `temporaryChatRetention: 1` (minimale Aufbewahrungsdauer)
- **24 Stunden**: `temporaryChatRetention: 24` (1 Tag)
- **168 Stunden**: `temporaryChatRetention: 168` (1 Woche)
- **720 Stunden**: `temporaryChatRetention: 720` (30 Tage – Standard)
- **8760 Stunden**: `temporaryChatRetention: 8760` (1 Jahr - Maximum)

## autoSubmitFromUrl

Steuert, ob ein über URL-Abfrageparameter auf `/c/new` bereitgestellter Prompt automatisch an das Modell übermittelt wird.

Wenn `/c/new?prompt=…&submit=true` von einem authentifizierten Benutzer geöffnet wird, füllt LibreChat normalerweise das Eingabefeld mit dem über die URL bereitgestellten Prompt vor und sendet diesen sofort ab. Dies ist eine Komfortfunktion für erstellte Deeplinks und geteilte Chat-URLs.

Für Bereitstellungen, bei denen Benutzer manipulierte Links aus externen Quellen erhalten könnten – und bei denen Modelle mit Speicher- oder Tool-Unterstützung sensible Kontextinformationen preisgeben könnten, falls ein Prompt-Injection-Payload das Modell erreicht –, können Betreiber die automatische Übermittlung deaktivieren. Wenn das Flag auf `false` gesetzt ist, wird der Prompt zwar weiterhin im Composer vorausgefüllt, der Benutzer muss jedoch explizit auf **Senden** klicken.

**Schlüssel:**

<OptionTable
  options={[
    [
      'autoSubmitFromUrl',
      'Boolean',
      'Steuert, ob `/c/new?prompt=…&submit=true` automatisch an das Modell übermittelt wird.',
      'When `false`, the prompt is pre-filled in the composer but not submitted.',
    ],
  ]}
/>

**Standard:** `true` (das bestehende Verhalten bleibt erhalten, sofern es nicht explizit deaktiviert wird).

**Hinweise:**

- Diese Einstellung hat keine Auswirkungen auf die URL-gesteuerte Auswahl von Model Specs oder andere URL-gesteuerte Einstellungen – nur auf den automatischen Übermittlungsschritt.
- Der Abfrageparameter akzeptiert sowohl `prompt` als auch `q` als Prompt-Quelle, wobei `prompt` Vorrang hat. `submit=true` ist der Auslöser.
- Empfohlen für Instanzen, die sensible Speicher- oder Tool-Daten verarbeiten, bei denen ein 1-Klick-Prompt-Injection-Vektor eine explizite Benutzerbestätigung erfordern sollte.

**Beispiel:**

```yaml filename="interface / autoSubmitFromUrl"
interface:
  autoSubmitFromUrl: false
```

## fileCitations

- Wenn Sie Agenten mit Dateisuchfunktion verwenden, kann das Zitierverhalten (Quantität und Qualität) über die [Agents endpoint configuration](/docs/configuration/librechat_yaml/object_structure/agents#file-citation-configuration-examples) konfiguriert werden.
