## fileStrategy

- **Optionen**: "local" | "firebase" | "s3" | "azure_blob" | "cloudfront"

<OptionTable
  options={[
    [
      'fileStrategy',
      'String',
      'Bestimmt, wo hochgeladene/generierte Benutzerdateien gespeichert werden. Standardmäßig auf `"local"` gesetzt, falls weggelassen.',
      'fileStrategy: "firebase"',
    ],
  ]}
/>

- **Hinweise**:
  - `"cloudfront"` speichert Dateien in S3 und gibt CloudFront-URLs für eine stabile Medienbereitstellung, signierte Cookies und signierte Downloads zurück.
  - `"firebase"` stellt Dateien über Firebase Storage und Firebase Hosting Edge-Standorte bereit.
  - S3 stellt Dateien über **presigned URLs** (temporäre signierte Token) bereit, die ablaufen. Sobald sie abgelaufen sind, wird jedes Bild oder jeder Avatar, der auf diese URL verweist, in der Benutzeroberfläche als defekt angezeigt. Dies macht S3 ungeeignet als primäre Strategie für visuelle Assets. Weitere Einzelheiten finden Sie in der [zugehörigen Diskussion](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903).
  - Für die beste Leistung von Bildern und Avataren verwenden Sie `"cloudfront"` oder `"firebase"`, oder konfigurieren Sie `fileStrategies`, um `avatar` und `image` an eine CDN-gestützte Strategie weiterzuleiten.
  - Bitte lesen Sie die [File Storage & CDN documentation](/docs/configuration/cdn) für Details zur Einrichtung.

## fileStrategies

Ermöglicht eine granulare Kontrolle über Dateispeicherstrategien für verschiedene Dateitypen.

- **Verfügbare Strategien**: "local" | "firebase" | "s3" | "azure_blob" | "cloudfront"

<OptionTable
  options={[
    [

      '',
    ],
  ]}
/>

**Untergeordnete Schlüssel:**

<OptionTable
  options={[
    [
      'default',
      'String',
      'Fallback-Speicherstrategie, wenn kein spezifischer Typ definiert ist. Standardmäßig "local".',
      '',
    ],
    [
      'avatar',
      'String',
      'Speicherstrategie für Benutzer- und Agenten-Avatarbilder. Für eine optimale Leistung wird eine CDN-gestützte Strategie (`"cloudfront"` oder `"firebase"`) empfohlen.',
      '',
    ],
    [
      'image',
      'String',
      'Speicherstrategie für in Chats hochgeladene Bilder. Für eine optimale Leistung wird die Verwendung einer CDN-gestützten Strategie (`"cloudfront"` oder `"firebase"`) empfohlen.',
      '',
    ],
    ['document', 'String', 'Speicherstrategie für Dokumenten-Uploads (PDFs, Textdateien usw.).', ''],
    ['skills', 'String', 'Speicherstrategie für Dateien, die mit Skills gebündelt sind.', ''],
  ]}
/>

- **Hinweise**:
  - Diese Einstellung hat Vorrang vor der einzelnen `fileStrategy`-Option
  - Wenn ein bestimmter Dateityp nicht konfiguriert ist, erfolgt ein Fallback auf `default`, dann auf `fileStrategy` und schließlich auf `"local"`.
  - Bilder und Avatare benötigen persistente, stabile URLs, um in der gesamten UI korrekt gerendert zu werden. S3-presigned URLs laufen ab (AWS-Limit: 7 Tage für IAM-Benutzer, Stunden für STS/rollenbasierte Anmeldeinformationen), was zu defekten Bildern in der Modellauswahl und der Chat-UI führt. Siehe die [zugehörige Diskussion](https://github.com/danny-avila/LibreChat/discussions/10280#discussioncomment-14803903) für den vollständigen Kontext. Verwenden Sie `"cloudfront"` oder `"firebase"` für `avatar` und `image`, um dies zu vermeiden.
  - S3 und Azure Blob Storage eignen sich gut für die `document`-Speicherung, bei der kurzlebige, vorab signierte Download-URLs angemessen sind.
  - Bitte lesen Sie die [File Storage & CDN documentation](/docs/configuration/cdn) für Einrichtungsdetails zu den einzelnen Speicheranbietern.

**Beispiele:**

```yaml filename="fileStrategies - All in one place"
# Use a single strategy for all file types
fileStrategies:
  default: 's3'
```

```yaml filename="fileStrategies - Mixed strategies"
# Route images and avatars to CDN, keep documents in object storage
fileStrategies:
  avatar: 'cloudfront' # CDN delivery for avatars
  image: 'cloudfront' # CDN delivery for generated/uploaded images
  document: 's3' # Object storage for documents
```

```yaml filename="fileStrategies - Partial configuration"
# Only configure specific types, others use default
fileStrategies:
  default: 'local'
  avatar: 'firebase' # Only avatars use Firebase CDN, everything else is local
```

## clientImageResize

**Schlüssel:**

<OptionTable
  options={[
    [
      'clientImageResize',
      'Object',
      'Konfiguriert die clientseitige Bildgrößenanpassung, um Datei-Uploads zu optimieren und Upload-Fehler aufgrund großer Bildgrößen zu verhindern.',

**Untergeordnete Schlüssel:**

<OptionTable
  options={[
    [
      'enabled',

      'Aktiviert oder deaktiviert die clientseitige Bildgrößenänderungsfunktion. Standard: false.',
      'enabled: true',
    ],
    [
      'maxWidth',
      'Number',
      'Maximale Breite in Pixeln für skalierte Bilder. Bilder, die breiter als dieser Wert sind, werden skaliert. Standard: 1920.',
      'maxWidth: 1024',
    ],
    [
      'maxHeight',
      'Number',
      'Maximale Höhe in Pixeln für skalierte Bilder. Bilder, die höher als dieser Wert sind, werden skaliert. Standard: 1080.',
      'maxHeight: 768',
    ],
    [
      'quality',
      'Number',
      'JPEG-Komprimierungsqualität (0.1 bis 1.0). Höhere Werte bedeuten bessere Qualität, aber größere Dateigrößen. Standard: 0.8.',
      'quality: 0.9',
    ],
    [
      'compressFormat',
      'String',
      'Ausgabeformat für komprimierte Bilder. Optionen: "jpeg", "webp". Standard: "jpeg".',
      'compressFormat: "webp"',
    ],
  ]}
/>

**Beschreibung:**

Die `clientImageResize`-Konfiguration ermöglicht die automatische Größenanpassung von Bildern auf der Client-Seite vor dem Hochladen. Diese Funktion hilft dabei:

- **Verhindern Sie Upload-Fehler**, die durch große Bilddateien entstehen, welche die Server-Limits überschreiten
- **Bandbreitennutzung reduzieren**, indem Bilder vor der Übertragung komprimiert werden
- **Verbessern Sie die Upload-Leistung** mit kleineren Dateigrößen
- **Bildqualität beibehalten** bei gleichzeitiger Optimierung der Dateigröße

Wenn aktiviert, werden Bilder, die die angegebenen `maxWidth`- oder `maxHeight`-Dimensionen überschreiten, automatisch auf der Client-Seite in der Größe angepasst, bevor sie auf den Server hochgeladen werden. Die Größenänderung behält das ursprüngliche Seitenverhältnis bei und stellt gleichzeitig sicher, dass das Bild innerhalb der angegebenen Grenzen bleibt.

**Beispiel:**

```yaml filename="clientImageResize"
fileConfig:
  clientImageResize:
    enabled: true
    maxWidth: 1920
    maxHeight: 1080
    quality: 0.8
    compressFormat: 'jpeg'
```

**Hinweise:**

- Gilt nur für Bilddateien (JPEG, PNG, WebP usw.)
- Die Größenänderung erfolgt automatisch, wenn Bilder die angegebenen Abmessungen überschreiten
- Das ursprüngliche Seitenverhältnis bleibt bei der Größenänderung erhalten
- Die Funktion funktioniert mit allen unterstützten Endpoints für den Bildupload.
- Die Qualitätseinstellung gilt nur für JPEG- und WebP-Formate
- Das Einstellen einer zu niedrigen Qualität (< 0.5) kann zu einer wahrnehmbaren Bildverschlechterung führen.

mcpSettings:
  # Strict whitelist mode:
  # allowedDomains:
  #   - "example.com"           # Specific domain
  #   - "*.example.com"         # All subdomains
  #   - "http://mcp-server:3000" # Internal service, explicitly whitelisted

  # Default SSRF mode with private service exemptions:
  allowedAddresses:
    - 'host.docker.internal:8080' # Permit one private host on one port
    - '10.0.0.5:8000' # Permit one private IP on one port

## mcpServers

**Schlüssel:**

<OptionTable
  options={[
    [
      'mcpServers',
      'Object',
      'Definiert die Konfiguration für Model Context Protocol (MCP) Server und ermöglicht die dynamische Integration von MCP-Servern innerhalb der Anwendung.',
      '',
    ],
  ]}
/>

**Untergeordnete Schlüssel:**

<OptionTable
  options={[
    [
      '<serverName>',
      'Object',
      'Jeder Schlüssel unter `mcpServers` repräsentiert eine individuelle MCP server Konfiguration, identifiziert durch einen eindeutigen Namen.',
      '',
    ],
  ]}
/>

- **Hinweise**:
  - Die Initialisierung erfolgt beim Start, und die App muss neu gestartet werden, damit Änderungen wirksam werden.
  - Der `<serverName>` ist ein eindeutiger Bezeichner für jede MCP Server-Konfiguration.
  - Jeder MCP-Server kann über einen von vier Verbindungstypen konfiguriert werden:
    - `stdio`
    - `websocket`
    - `sse`
    - `streamable-http`
  - Das Feld `type` gibt den Verbindungstyp zum MCP Server an.
  - Wenn `type` weggelassen wird, erfolgt die Standardeinstellung basierend auf dem Vorhandensein und dem Format von `url` oder `command`:
    - Wenn `url` angegeben ist und mit `http` oder `https` beginnt, ist der Standardwert für `type` `sse`.
    - Wenn `url` angegeben ist und mit `ws` oder `wss` beginnt, wird `type` standardmäßig auf `websocket` gesetzt.
    - Wenn `command` angegeben ist, ist der Standardwert für `type` `stdio`.
  - Zusätzliche Konfigurationsoptionen umfassen:
    - `timeout`: Timeout in Millisekunden für MCP-Server-Anfragen. Bestimmt, wie lange auf eine Antwort für Tool-Anfragen gewartet werden soll.
    - `initTimeout`: Timeout in Millisekunden für die Initialisierung des MCP-Servers. Bestimmt, wie lange auf die Initialisierung des Servers gewartet werden soll.
    - `serverInstructions`: Steuert, ob Server-Anweisungen in den Agenten-Kontext einbezogen werden. Kann `true` (vom Server bereitgestellt), `false` (deaktiviert) oder eine benutzerdefinierte Zeichenfolge (überschreibt die vom Server bereitgestellten) sein.
    - `customUserVars`: (Optional) Definiert benutzerdefinierte Variablen (z. B. API-Schlüssel, URLs), die einzelne Benutzer für einen MCP-Server festlegen können. Diese benutzerspezifischen Werte, die über die Benutzeroberfläche bereitgestellt werden, können dann in den `headers`- oder `env`-Konfigurationen des Servers unter Verwendung der `{{VARIABLE_NAME}}`-Syntax referenziert werden. Dies ermöglicht eine benutzerspezifische Authentifizierung oder Anpassung für MCP-Tools.
  - siehe: [MCP Servers Object Structure](/docs/configuration/librechat_yaml/object_structure/mcp_servers)

**Beispiel:**

```yaml filename="mcpServers"
mcpServers:
  everything:
    # type: sse # type can optionally be omitted
    url: http://localhost:3001/sse
    timeout: 30000
    initTimeout: 10000
    serverInstructions: true # Use server-provided instructions
  puppeteer:
    type: stdio
    command: npx
    args:
      - -y
      - '@modelcontextprotocol/server-puppeteer'
    timeout: 30000
    initTimeout: 10000
    serverInstructions: 'Do not access any local files or local/internal IP addresses'
  filesystem:
    # type: stdio
    command: npx
    args:
      - -y
      - '@modelcontextprotocol/server-filesystem'
      - /home/user/LibreChat/
    iconPath: /home/user/LibreChat/client/public/assets/logo.svg
  mcp-obsidian:
    command: npx
    args:
      - -y
      - 'mcp-obsidian'
      - /path/to/obsidian/vault
  streamable-http-example:
    type: streamable-http
    url: https://example.com/mcp
    headers:
      Authorization: 'Bearer ${API_TOKEN}'
    timeout: 30000
  per-user-crendentials-example:
    type: sse
    url: 'https//some.mcp/sse'
    headers:
      X-Custom-Auth-Token: '{{USER_API_KEY}}' # Placeholder for the user-provided API key, defined in `customUserVars` below.
    customUserVars:
      USER_API_KEY:
        title: 'Service API Key'
        description: "Your personal API key for this service. You can get it <a href='https://example.com/api-keys' target='_blank'>here</a>."
    serverInstructions: true
```
