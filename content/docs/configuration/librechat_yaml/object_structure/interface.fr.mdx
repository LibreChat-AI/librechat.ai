## Vue d'ensemble [#overview]

L'objet `interface` permet la personnalisation de divers éléments de l'interface utilisateur au sein de l'application, incluant les paramètres de visibilité et de comportement pour des composants tels que les menus, les panneaux et les liens. Cette section fournit une analyse détaillée de la structure de l'objet `interface`.

Voici les champs sous `interface` :

- `mcpServers`
- `privacyPolicy`
- `termsOfService`
- `modelSelect`
- `parameters`
- `contextUsage`
- `contextCost`
- `currency`
- `presets`
- `prompts`
- `bookmarks`
- `memories`
- `multiConvo`
- `agents`
- `remoteAgents`
- `skills`
- `sharedLinks`
- `temporaryChat`
- `temporaryChatRetention`
- `retentionMode`
- `retainAgentFiles`
- `autoSubmitFromUrl`
- `customWelcome`
- `runCode`
- `webSearch`
- `fileSearch`
- `fileCitations`
- `defaultPinnedTools`
- `peoplePicker`
- `marketplace`

**Notes :**

- Les configurations `interface` sont appliquées globalement au sein de l'application.
- Des valeurs par défaut sont fournies pour la plupart des paramètres, mais elles peuvent être remplacées en fonction de besoins ou de conditions spécifiques.
- La logique conditionnelle dans l'application peut modifier davantage ces paramètres en fonction d'autres configurations telles que les Model Specs.

<Callout type="warning" title="Obsolète : champs d'effets secondaires de permission">
Plusieurs champs ci-dessous (`mcpServers`, `prompts`, `bookmarks`, `memories`, `multiConvo`, `agents`, `remoteAgents`, `skills`, `sharedLinks`, `temporaryChat`, `runCode`, `webSearch`, `fileSearch`, `fileCitations`, `peoplePicker`, `marketplace`) ne se contentent pas de basculer l'interface utilisateur, ils initialisent les permissions de rôle dans la base de données au démarrage, et ce uniquement pour le rôle `USER` intégré.

Pour une gestion continue, utilisez le [**LibreChat Admin Panel**](/docs/features/admin_panel), qui modifie directement la matrice des permissions sur chaque rôle (y compris les rôles personnalisés). Ces champs YAML restent pris en charge pour l'initialisation d'une nouvelle instance ou pour les déploiements entièrement pilotés par fichier, mais ne devraient plus être utilisés comme méthode principale pour gérer les permissions des fonctionnalités.

Consultez [Access Control](/docs/features/access_control) pour le modèle de permission complet.

</Callout>

## Exemple [#example]

```yaml filename="interface"
interface:
  mcpServers:
    placeholder: 'MCP Servers'
    use: true
    create: true
    share: false
    public: false
    configureObo: false
    trustCheckbox:
      label: 'I trust this server'
      subLabel: 'Only enable servers you trust'
  privacyPolicy:
    externalUrl: 'https://example.com/privacy'
    openNewTab: true
  termsOfService:
    externalUrl: 'https://example.com/terms'
    openNewTab: true
    modalAcceptance: true
    modalTitle: 'Terms of Service'
    modalContent: |
      # Terms of Service
      ## Introduction
      Welcome to LibreChat!
  modelSelect: false
  parameters: true
  contextUsage: true
  contextCost: true
  currency:
    code: EUR
    rate: 0.92
  presets: false
  prompts:
    use: true
    create: true
    share: true
    public: false
  bookmarks: true
  multiConvo: true
  agents:
    use: true
    create: true
    share: true
    public: false
  skills:
    use: true
    create: true
    share: false
    public: false
    defaultActiveOnShare: false
  sharedLinks:
    create: true
    share: true
    public: false
    snapshotFiles: true
  customWelcome: 'Hey {{user.name}}! Welcome to LibreChat'
  runCode: true
  webSearch: true
  fileSearch: true
  fileCitations: true
  defaultPinnedTools:
    - artifacts
    - execute_code
    - mcp
```

## prompts

> **Obsolète pour la gestion des permissions.** Initialise les permissions de rôle `PROMPTS` au démarrage uniquement pour le rôle `USER` par défaut. Préférez le [Panneau d'administration](/docs/features/admin_panel) pour gérer les permissions des prompts par rôle/groupe/utilisateur.

**Clé :**

<OptionTable
  options={[
    [
      'prompts',
      'Boolean or Object',
      'Contrôle les fonctionnalités liées aux prompts pour tous les utilisateurs. Peut être un booléen pour une activation/désactivation simple, ou un objet pour un contrôle granulaire sur l\'utilisation, la création, le partage et la visibilité publique.',
      'When set to `false`, users will not have access to create, edit, or use custom prompts.',
    ],
  ]}
/>

**Par défaut :** `true`

**Important : Configuration booléenne vs objet**

- **Booléen (`prompts: true`)** : Met uniquement à jour l'autorisation `use`. Les valeurs d'autorisation `create`, `share` et `public` existantes sont **préservées** dans la base de données. Utilisez ceci comme un simple commutateur de fonctionnalité sans affecter les autres paramètres configurés via le panneau d'administration.

- **Objet** : Met à jour uniquement les sous-autorisations explicitement spécifiées. Toutes les autorisations non incluses dans la configuration sont conservées telles qu'elles sont dans la base de données.

Lors de l'utilisation de la structure d'objet :

## temporaryChatRetention

La configuration `temporaryChatRetention` vous permet de personnaliser la durée de conservation des conversations temporaires avant qu'elles ne soient automatiquement supprimées.

**Clé :**

<OptionTable
  options={[

    ],
  ]}
/>

**Règles de validation :**

- **Minimum** : 1 heure (empêche la suppression immédiate)
- **Maximum** : 8760 heures (rétention maximale de 1 an)
- **Par défaut** : 720 heures (30 jours)

**Méthodes de configuration :**

1. **LibreChat.yaml** (recommandé) : `interface.temporaryChatRetention: 168`
2. **Variable d'environnement** (obsolète) : `TEMP_CHAT_RETENTION_HOURS=168`

> **Note :** La variable d'environnement `TEMP_CHAT_RETENTION_HOURS` est obsolète. Veuillez utiliser l'option de configuration `interface.temporaryChatRetention` dans `librechat.yaml` à la place. La valeur du fichier de configuration est prioritaire sur la variable d'environnement.

**Exemple :**

```yaml filename="interface / temporaryChatRetention"
interface:
  temporaryChatRetention: 168 # Retain temporary chats for 7 days
  retentionMode: 'temporary'
```

**Périodes de rétention courantes :**

- **1 heure** : `temporaryChatRetention: 1` (rétention minimale)
- **24 heures** : `temporaryChatRetention: 24` (1 jour)
- **168 heures** : `temporaryChatRetention: 168` (1 semaine)
- **720 heures** : `temporaryChatRetention: 720` (30 jours - par défaut)
- **8760 heures** : `temporaryChatRetention: 8760` (1 an - maximum)

## autoSubmitFromUrl

Détermine si une invite fournie via les paramètres de requête d'URL sur `/c/new` est automatiquement soumise au modèle.

Lorsqu'un utilisateur authentifié ouvre `/c/new?prompt=…&submit=true`, LibreChat pré-remplit normalement le champ de saisie avec le prompt fourni dans l'URL et le soumet immédiatement. Il s'agit d'une fonctionnalité pratique pour les liens profonds personnalisés et les URL de chat partagées.

Pour les déploiements où les utilisateurs peuvent recevoir des liens conçus par des sources externes — et où les modèles activés pour la mémoire ou les outils pourraient divulguer un contexte sensible si une charge utile d'injection de prompt atteint le modèle — les opérateurs peuvent désactiver la soumission automatique. Avec cet indicateur défini sur `false`, le prompt est toujours pré-rempli dans le compositeur, mais l'utilisateur doit appuyer explicitement sur **Envoyer**.

**Clé :**

<OptionTable
  options={[
    [
      'autoSubmitFromUrl',
      'Boolean',
      'Détermine si `/c/new?prompt=…&submit=true` envoie automatiquement la requête au modèle.',
      'When `false`, the prompt is pre-filled in the composer but not submitted.',
    ],
  ]}
/>

**Par défaut :** `true` (le comportement existant est préservé à moins d'être explicitement désactivé).

**Notes :**

- Ce paramètre n'affecte pas la sélection de Model Specs basée sur l'URL ou d'autres paramètres basés sur l'URL — seulement l'étape de soumission automatique.
- Le paramètre de requête accepte à la fois `prompt` et `q` comme source d'invite, `prompt` étant prioritaire. `submit=true` est le déclencheur.
- Recommandé pour les instances traitant des données sensibles de mémoire ou d'outils, où un vecteur d'injection de prompt en un clic devrait nécessiter une confirmation explicite de l'utilisateur.

**Exemple :**

```yaml filename="interface / autoSubmitFromUrl"
interface:
  autoSubmitFromUrl: false
```
