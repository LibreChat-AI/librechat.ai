[OpenRouter](https://openrouter.ai/) fournit un accès à des centaines de modèles d'IA via une API unique. Ce guide vous accompagne dans la configuration d'OpenRouter en tant qu'endpoint personnalisé dans LibreChat, étape par étape.

<Callout type="info" title="Prérequis">

Avant de commencer, assurez-vous d'avoir :
- LibreChat installé et en cours d'exécution (voir [configuration Docker](/docs/local/docker))
- Un fichier `librechat.yaml` créé et monté (voir le [guide librechat.yaml](/docs/configuration/librechat_yaml))

</Callout>

## Configuration [#setup]

<Steps>
  <Step>

### Obtenir une clé API [#get-an-api-key]

Créez un compte sur [openrouter.ai](https://openrouter.ai/) et générez une clé API depuis la [page Keys](https://openrouter.ai/keys).

Copiez la clé -- elle commence par `sk-or-v1-`.

  </Step>
  <Step>

### Ajoutez la clé à votre fichier .env [#add-the-key-to-your-env-file]

Ouvrez votre fichier `.env` à la racine du projet et ajoutez votre clé API OpenRouter :

```bash filename=".env"
OPENROUTER_KEY=sk-or-v1-your-key-here
```

<Callout type="error" title="Utilisez OPENROUTER_KEY, pas OPENROUTER_API_KEY">

Vous devez utiliser `OPENROUTER_KEY` comme nom de variable. L'utilisation de `OPENROUTER_API_KEY` remplacera également l'endpoint OpenAI intégré pour utiliser OpenRouter, ce qui n'est presque certainement pas ce que vous souhaitez.

</Callout>

  </Step>
  <Step>

### Ajouter l'endpoint à librechat.yaml [#add-the-endpoint-to-librechatyaml]

Ajoutez ce qui suit à votre fichier `librechat.yaml`. Si le fichier contient déjà du contenu, fusionnez la section `endpoints` avec votre configuration existante :

```yaml filename="librechat.yaml"
version: 1.3.5
cache: true
endpoints:
  custom:
    - name: "OpenRouter"
      apiKey: "${OPENROUTER_KEY}"
      baseURL: "https://openrouter.ai/api/v1"
      models:
        default: ["meta-llama/llama-3-70b-instruct"]
        fetch: true
      titleConvo: true
      titleModel: "meta-llama/llama-3-70b-instruct"
      dropParams: ["stop"]
      modelDisplayLabel: "OpenRouter"
```

Explication des champs clés :

| Champ | Objectif |
|-------|---------|
| `apiKey: "${OPENROUTER_KEY}"` | Référence la variable d'environnement de l'étape 2. La syntaxe `${}` indique à LibreChat de lire la valeur depuis `.env`. |
| `models.fetch: true` | Récupère la liste complète des modèles depuis l'API d'OpenRouter, afin que les nouveaux modèles apparaissent automatiquement. |
| `dropParams: ["stop"]` | Supprime le paramètre `stop` des requêtes. Les modèles OpenRouter utilisent des jetons d'arrêt variés, donc le supprimer évite des problèmes de compatibilité. |
| `modelDisplayLabel: "OpenRouter"` | Le nom affiché dans le sélecteur d'endpoint de LibreChat. |

</Step>
  <Step>

### Redémarrer LibreChat [#restart-librechat]

<Tabs items={['Docker', 'Local']}>
  <Tabs.Tab>

```bash
docker compose down && docker compose up -d
```

  </Tabs.Tab>
  <Tabs.Tab>

Arrêtez le processus en cours (Ctrl+C) et redémarrez :

```bash
npm run backend
```

  </Tabs.Tab>
</Tabs>

  </Step>
  <Step>

### Vérifier que cela fonctionne [#verify-it-works]

Ouvrez LibreChat dans votre navigateur. Vous devriez voir **OpenRouter** dans le menu déroulant de sélection de l'endpoint. Sélectionnez-le pour voir les modèles disponibles.

Si OpenRouter n'apparaît pas, vérifiez les journaux du serveur pour détecter d'éventuelles erreurs de configuration :

```bash
docker compose logs api | grep -i "error\|openrouter"
```

  </Step>
</Steps>

## Dépannage [#troubleshooting]

### OpenRouter n'apparaît pas dans le sélecteur de modèles [#openrouter-does-not-appear-in-the-model-selector]

Vérifiez ces éléments dans l'ordre :

1. `OPENROUTER_KEY` est défini dans `.env`.
2. `librechat.yaml` contient l'entrée OpenRouter sous `endpoints.custom`.
3. Les utilisateurs de Docker ont monté `librechat.yaml` dans `docker-compose.override.yml`.
4. LibreChat a été redémarré après la modification des deux fichiers.
5. Les journaux de l'API n'affichent pas les erreurs de validation YAML.

```bash
docker compose logs api
```

Si les journaux mentionnent une variable d'environnement manquante, vérifiez que le nom de la variable dans `.env` correspond à la variable référencée dans `librechat.yaml`. Pour ce guide, cela signifie `apiKey: "${OPENROUTER_KEY}"` dans `librechat.yaml` et `OPENROUTER_KEY=...` dans `.env`.

### OpenRouter renvoie 402 Payment Required [#openrouter-returns-402-payment-required]

Une réponse `402 Payment Required` provient d'OpenRouter, et non de LibreChat. Ajoutez des crédits ou choisissez un modèle gratuit/disponible dans OpenRouter, puis réessayez le même endpoint dans LibreChat.

## Personnalisation [#customization]

### Utilisation d'une clé API user_provided [#using-user_provided-api-key]

Au lieu de stocker la clé dans `.env`, vous pouvez laisser chaque utilisateur fournir sa propre clé via l'interface utilisateur de LibreChat :

```yaml
apiKey: "user_provided"
```

Les utilisateurs verront un champ de saisie de clé lors de la sélection de l'endpoint OpenRouter.

### Limiter les modèles disponibles [#limiting-available-models]

Au lieu de récupérer tous les modèles, vous pouvez spécifier une liste fixe :

```yaml
models:
  default: ["anthropic/claude-3.5-sonnet", "openai/gpt-4o", "meta-llama/llama-3-70b-instruct"]
  fetch: false
```
