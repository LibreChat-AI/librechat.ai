## 什么是 librechat.yaml？ [#what-is-librechatyaml]

`librechat.yaml` 文件是 LibreChat 用于自定义 AI endpoint、模型设置、界面选项以及 MCP 服务器和智能体等高级功能的主要配置文件。它是可选的——如果该文件不存在，LibreChat 将使用合理的默认设置运行。

请按照以下步骤创建文件，针对您的部署类型进行挂载，并验证其是否正常工作。

<Callout type="info" title="如果你只能记住一件事">

对于 Docker 安装，仅编辑 `librechat.yaml` 是不够的。该文件必须存在于项目根目录中，挂载到 API 容器内，并且必须重启 LibreChat，更改才会显示在 UI 中。

</Callout>

<Callout type="info" title="更喜欢使用 UI？请使用管理面板">

[**LibreChat Admin Panel**](/docs/features/admin_panel) 可在浏览器中管理相同的配置，包括在登录时生效且无需重启 LibreChat 的按角色和按组的覆盖设置。它随官方 Docker Compose 堆栈一同提供。请使用 `librechat.yaml` 进行基于文件的配置或引导设置，并使用管理面板进行日常管理。

</Callout>

## 设置 [#setup]

<Steps>
  <Step>

### 找到或创建文件 [#locate-or-create-the-file]

在您的项目根目录（与您的 `.env` 文件相同的目录）中创建一个新的 `librechat.yaml`：

```bash
touch librechat.yaml
```

您也可以复制 [示例配置](/docs/configuration/librechat_yaml/example) 作为起点：

```bash
cp librechat.example.yaml librechat.yaml
```

<Callout type="info" title="备选文件路径">

您可以使用 `CONFIG_PATH` 环境变量设置自定义文件路径：

```bash filename=".env"
CONFIG_PATH="/alternative/path/to/librechat.yaml"
```

</Callout>

  </Step>
  <Step>

### 挂载配置文件 [#mount-the-config-file]

<Tabs items={['Docker', '本地']}>
  <Tabs.Tab>

Docker 需要一个卷挂载来访问容器内的 `librechat.yaml` 文件。

**复制示例覆盖文件：**

```bash
cp docker-compose.override.yml.example docker-compose.override.yml
```

**编辑 `docker-compose.override.yml`** 并确保 librechat.yaml 的卷挂载（volume mount）已取消注释：

```yaml filename="docker-compose.override.yml"
services:
  api:
    volumes:
      - type: bind
        source: ./librechat.yaml
        target: /app/librechat.yaml
```

这使用了 [docker-compose.override.yml](/docs/configuration/docker_override) 模式——Docker Compose 会自动将其与主 `docker-compose.yml` 合并，因此您的自定义配置在更新后依然有效。

  </Tabs.Tab>
  <Tabs.Tab>

将 `librechat.yaml` 放置在项目根目录（与您的 `.env` 文件相同的目录）中。对于本地安装，无需进行额外的挂载。

  </Tabs.Tab>
</Tabs>

  </Step>
  <Step>

### 重启 LibreChat [#restart-librechat]

<Tabs items={['Docker', '本地']}>
  <Tabs.Tab>

```bash
docker compose down && docker compose up -d
```

  </Tabs.Tab>
  <Tabs.Tab>

停止正在运行的进程 (Ctrl+C) 并重启：

```bash
npm run backend
```

  </Tabs.Tab>
</Tabs>

  </Step>
  <Step>

### 验证其是否正常工作 [#verify-it-works]

在浏览器中打开 LibreChat。如果你的配置中包含自定义 endpoint，你应该能在模型选择下拉菜单中看到它们。

如果服务器启动失败，请检查日志以获取验证错误信息：

```bash
docker compose logs api
```

  </Step>
</Steps>

## 示例：添加 OpenRouter [#example-adding-openrouter]

此示例将引导您完成将 [OpenRouter](https://openrouter.ai/) 添加为自定义 endpoint 的过程——这是最受欢迎的配置之一。

**1. 获取 API 密钥**，请访问 [openrouter.ai/keys](https://openrouter.ai/keys)。

**2. 将该密钥添加到您的 `.env` 文件中：**

```bash filename=".env"
OPENROUTER_KEY=sk-or-v1-your-key-here
```

<Callout type="warning" title="环境变量名称">

请使用 `OPENROUTER_KEY`，而不是 `OPENROUTER_API_KEY`。使用 `OPENROUTER_API_KEY` 会导致 OpenAI endpoint 也被覆盖并使用 OpenRouter。

</Callout>

**3. 将 endpoint 添加到 `librechat.yaml`：**

```yaml filename="librechat.yaml"
version: 1.3.5
cache: true
endpoints:
  custom:
    - name: "OpenRouter"
      apiKey: "${OPENROUTER_KEY}"
      baseURL: "https://openrouter.ai/api/v1"
      models:
        default: ["meta-llama/llama-3-70b-instruct"]
        fetch: true
      titleConvo: true
      titleModel: "meta-llama/llama-3-70b-instruct"
      dropParams: ["stop"]
      modelDisplayLabel: "OpenRouter"
```

**4. 重启 LibreChat**（请参阅上方的重启命令），然后从模型选择器中选择 OpenRouter。

有关包含更多 endpoint 示例的完整带注释配置文件，请参阅[示例配置](/docs/configuration/librechat_yaml/example)。

## 故障排除 [#troubleshooting]

### 更改未在 LibreChat 中显示 [#change-does-not-show-in-librechat]

如果您编辑了 `librechat.yaml` 但 UI 中没有任何变化：

1. 除非您设置了 `CONFIG_PATH`，否则请确认该文件位于 LibreChat 项目根目录下。
2. 对于 Docker，请确认该文件已在 `docker-compose.override.yml` 中挂载。
3. 使用 `docker compose down && docker compose up -d` 重启 LibreChat。
4. 使用 `docker compose logs api` 查看 API 日志。
5. 使用 [YAML Validator](/toolkit/yaml_checker) 验证该文件。

像 OpenRouter 这样的自定义 endpoint 只有在以下三个条件全部满足时才会显示：`.env` 包含密钥，`librechat.yaml` 定义了该 endpoint，并且 Docker 能够读取挂载的配置文件。

### 配置验证 [#configuration-validation]

<Callout type="error" title="配置验证">

如果 `librechat.yaml` 包含验证错误，LibreChat 将会报错并退出（退出代码 1）。这种快速失败（fail-fast）机制可以尽早发现配置问题。

在重启之前，请使用 [YAML Validator](/toolkit/yaml_checker) 或 [yamlchecker.com](https://yamlchecker.com/) 来验证您的 YAML 语法。

</Callout>

### 服务器启动后立即退出 [#server-exits-immediately-on-startup]

如果您的服务器在启动后立即退出，这很可能是配置验证错误导致的。

**诊断方法：**

1. 检查服务器日志：`docker compose logs api`
2. 使用 [YAML Validator](/toolkit/yaml_checker) 验证您的 YAML 语法
3. 常见错误：缩进不正确、缺少冒号、未知键、无效值

**临时变通方案**（不建议在生产环境中使用）：

```bash filename=".env"
CONFIG_BYPASS_VALIDATION=true
```

<Callout type="warning" title="警告">

`CONFIG_BYPASS_VALIDATION=true` 会导致服务器跳过验证并使用默认配置。请务必修复验证错误，而不是使用此选项。

</Callout>
